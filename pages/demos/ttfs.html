Time-to-first-sizing — image-element vs auto strategy

Generates a large PNG (4000×3000 as a Blob), then calls prepare() with each strategy and measures how long it takes to return dimensions. The auto strategy (default) byte-probes the first ~24 bytes and returns; image-element waits for a full decode.

What this shows

Both API calls return the same {width, height}. Only the pipeline differs. Here the image lives in-memory as a Blob, so the network term is zero; the delta is pure decode vs byte-parse.

For remote URLs the default strategy also streams the fetch — dimensions are returned after the first ~2KB arrive, so the full transfer doesn't have to complete before layout can proceed.

Headline numbers

5MB JPEG over 10 Mbps:
  image-element strategy:  ~4000ms
  default (auto):          ~150ms

4000×3000 PNG Blob:
  image-element strategy:  ~80ms  (decode only)
  default (auto):          ~5ms   (probe IHDR)

When the probe path yields

AVIF, HEIC, or truncated bytes trigger a transparent createImageBitmap fallback on the bytes we already streamed — still faster than a round trip because the bytes are buffered, just without the first-2KB shortcut.
